### 第六阶段详细架构

#### 微信订阅消息授权积分系统

```
用户授权(requestSubscribeMessage) → 前端上报积分(/api/wx-subscribe/authorize)
    → 后端WxSubscribeAuthorizationService.addCredits(userId, templateType, 1)
    → wx_subscribe_authorization表 credits + 1

定时任务/消息消费 → WxSubscribeMessageService.sendSubscribeMessage()
    → WxSubscribeAuthorizationService.consumeCredit(userId, templateType)
    → credits - 1, used_credits + 1 (原子SQL)
    → 调用微信API发送消息
```

#### RabbitMQ延迟消息提醒系统

```
┌─────────────────────────────────────────────────────────────────┐
│                     消息生产（写入路径）                           │
│                                                                 │
│  用户创建/修改喂食提醒 → FeedingReminderService                   │
│       ↓ scheduleFeedingReminder()                               │
│  计算延迟时间(到下次提醒的毫秒数)                                   │
│       ↓                                                         │
│  ReminderMessageProducer.sendDelayedFeedingReminder(msg, delay) │
│       ↓                                                         │
│  RabbitMQ: x-delayed-message Exchange → 延迟投递到 Queue          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     消息消费（读取路径）                           │
│                                                                 │
│  RabbitMQ到期投递 → ReminderMessageConsumer.onFeedingReminder()  │
│       ↓                                                         │
│  校验DB中提醒是否仍启用（防脏消息）                                 │
│       ↓                                                         │
│  发送站内信 + 微信订阅消息（消费授权积分）                           │
│       ↓                                                         │
│  自动调度下次提醒 → scheduleNextFeedingReminder()                 │
│       ↓                                                         │
│  ReminderMessageProducer.sendDelayedFeedingReminder(msg, delay) │
└─────────────────────────────────────────────────────────────────┘
```

**RabbitMQ Exchange/Queue 设计：**

| 组件 | 名称 | 类型 | 说明 |
|------|------|------|------|
| Exchange | reminder.delayed.exchange | x-delayed-message | 延迟消息交换机 |
| Queue | reminder.feeding.queue | durable | 喂食提醒队列 |
| Queue | reminder.checkin.queue | durable | 打卡提醒队列 |
| Queue | reminder.feeding.queue.dlq | durable | 喂食提醒死信队列 |
| Queue | reminder.checkin.queue.dlq | durable | 打卡提醒死信队列 |
| Routing Key | reminder.feeding | - | 喂食路由键 |
| Routing Key | reminder.checkin | - | 打卡路由键 |

**各业务提醒触发方式对比：**

| 业务类型 | 触发方式 | 说明 |
|---------|---------|------|
| 疫苗提醒 | @Scheduled 每天9:00 | 检查7天内到期，一次性通知 |
| 驱虫提醒 | @Scheduled 每天9:00 | 检查7天内到期，一次性通知 |
| 打卡提醒 | RabbitMQ 延迟消息 | 创建时调度，消费后自动调度下次 |
| 喂食提醒 | RabbitMQ 延迟消息 | 创建时调度，消费后自动调度下次，支持weekday/weekend过滤 |

**性能对比（vs 旧方案）：**

| 维度 | 定时轮询方案 | Redis缓存方案 | RabbitMQ方案 |
|------|------------|-------------|-------------|
| 每分钟DB查询 | 2次全表扫描 | 0次 | 0次 |
| 每分钟Redis查询 | 0次 | 2次GET | 0次 |
| 消息投递精度 | 1分钟 | 1分钟 | 毫秒级 |
| 重复提醒调度 | 需要额外逻辑 | 需要额外逻辑 | 消费后自动调度下次 |
| 脏消息处理 | 无 | 缓存可能不一致 | 消费时校验DB状态 |
| 扩展性 | 差（N用户=N倍查询） | 中（Redis内存限制） | 优（MQ天然支持高并发） |

## 第八阶段：后端数据逻辑修复 🔄

### 问题分析

动态(posts)表的 `status` 和 `deleted` 字段语义混淆，导致删除逻辑异常：

| 字段 | 数据库定义 | 实际使用 |
|------|-----------|---------|
| `status` | 0-审核中 1-正常 2-删除 | 用户删除时设为0（与"审核中"冲突） |
| `deleted` | 0-未删除 1-已删除 | 仅管理员删除时使用 |

**核心问题**：
1. 用户删除动态设置 `status=0`，与"审核中"语义冲突，导致已删除动态被定时审核任务重新审核通过
2. 管理员删除设置 `deleted=1`，但所有查询只过滤 `status=1` 不检查 `deleted`，管理员删除对用户不可见
3. PostMapper.xml 的 ResultMap 缺少 `deleted`、`audit_status`、`audit_remark` 字段映射
4. `selectPendingForAudit` 用 `status=0` 查询待审核，会误查到用户删除的动态
5. `adminListPosts` 未过滤 `deleted=1`，管理员列表显示已删除动态
6. `adminRestorePost` 只恢复 `deleted`，不恢复 `status`

| # | 任务 | 状态 | 优先级 | 变更说明 |
|---|------|------|--------|---------|
| 40 | 修复动态删除字段混用 | ✅ 完成 | P0 | 用户删除从 `setStatus(0)` 改为 `setDeleted(1)`，统一用户删除和管理员删除使用同一字段；创建动态时增加 `setAuditStatus(0)` |
| 41 | 修复 PostMapper.xml 查询条件 | ✅ 完成 | P0 | 所有9个查询增加 `AND deleted = 0`；`selectPendingForAudit` 从 `status=0` 改为 `audit_status=0`；ResultMap 补充 `audit_status`/`audit_remark`/`deleted` 映射 |
| 42 | 修复管理员恢复和列表逻辑 | ✅ 完成 | P1 | `adminRestorePost` 同时恢复 `deleted=0` 和 `status=1`；`adminListPosts` 增加 `deleted=0` 过滤；`getUserPostCount` 增加 `deleted=0` 条件 |
