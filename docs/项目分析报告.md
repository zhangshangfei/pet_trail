## 三、性能优化点

### 🔴 高优先级

#### 1. CommentService N+1 查询问题
- CommentService.convertToVO() 每条评论单独查询用户信息
- 每条回复也单独查询用户信息
- 20条评论可能产生 40+ 次数据库查询
- 建议：批量预加载用户信息（与 PostController 的 convertToPostVOList 同样模式）

#### ~~2. RecommendService 全量加载问题~~ ✅ 已优化
- **原问题**：RecommendService.computeRecommendations() 查询所有未关注用户计算评分
- **已修复**：动态推荐候选集限制为近30天500条（selectCandidatePosts），用户推荐已有Redis缓存
- 作者品种/类别信息带缓存（authorBreedCache/authorCategoryCache），同一作者只查一次

#### 3. PostMapper.xml ResultMap 字段缺失
- 缺少 videos、ee_count、location、stickers、bubble 映射
- 可能导致部分查询场景数据丢失

### 🟡 中优先级

#### 4. 图片上传未压缩
- 发布动态时直接上传原图
- 应在上传前进行压缩（尤其是手机拍摄的高分辨率照片）
- 建议：使用 uni.compressImage 压缩后再上传

#### 5. 通知列表无分页优化
- NotificationService 使用 LIMIT offset,size 分页
- 数据量大时深分页性能差
- 建议：使用游标分页（基于 last_id）

#### 6. 缺少 API 限流
- 发布动态、点赞、评论等接口无频率限制
- 恶意用户可刷接口
- 建议：使用 Redis + 令牌桶/滑动窗口限流

#### 7. 前端请求缓存粒度粗
- 当前 3 秒 TTL 的全局缓存
- 对实时性要求高的接口（点赞状态）不应缓存
- 建议：按接口类型设置不同缓存策略

### 🟢 低优先级

#### 8. 前端无全局错误处理
- 各页面独立 try-catch
- 建议：统一请求拦截器处理 401/403/500

#### 9. 缺少骨架屏
- 列表加载时只有"加载中..."文字
- 建议：添加骨架屏提升感知速度

#### 10. 无单元测试
- 后端无任何测试类
- 建议：至少为 Service 层核心逻辑添加测试

## 六、推荐算法详细文档

### 6.1 用户推荐算法（RecommendService.computeRecommendations）

用于「发现用户」页面的推荐Tab，基于5维评分模型推荐可能感兴趣的用户。

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| 品种匹配 | 0.30 | 同品种1.0，同类别0.6，无关0.1 |
| 活跃度 | 0.25 | `min(1, 粉丝数/50) × 0.6 + min(1, 动态数/20) × 0.4` |
| 社交距离 | 0.20 | 二度人脉1.0，否则0.0 |
| 互动亲密度 | 0.15 | 点赞过同一动态的用户1.0，否则0.0 |
| 新用户加权 | 0.10 | 3天内1.0，7天内0.8，14天内0.5，30天内0.3 |

**缓存策略**：Redis 缓存完整推荐列表，TTL 2小时

### 6.2 动态推荐算法（RecommendService.computePostRecommendations） ✅ 新增

用于首页「推荐」Tab，基于5维评分模型个性化推荐动态内容。

| 维度 | 权重 | 计算方式 |
|------|------|---------|
| 兴趣匹配 | 0.30 | 作者宠物品种/类别与当前用户相似度（同品种1.0，同类别0.6，无关0.1） |
| 热度衰减 | 0.25 | `(点赞×3 + 评论×2 + 分享) / (小时数+2)`，归一化到0~1（÷10取min） |
| 社交距离 | 0.20 | 二度人脉1.0，已关注0.5，无关系0.0 |
| 互动亲密度 | 0.15 | 曾点赞过该作者的其他动态 → 1.0，否则0.0 |
| 新鲜度 | 0.10 | 1小时内1.0，6小时内0.9，24小时内0.7，72小时内0.5，7天内0.3，15天内0.1 |

**降级策略**：未登录用户使用纯热度排序SQL（selectRecommendFeed）

**候选集限制**：仅取近30天动态，最多500条（selectCandidatePosts），避免全量计算

**缓存策略**：
- 推荐结果缓存：Redis `recommend:posts:{userId}`，TTL 30分钟
- 作者品种/类别信息带内存缓存（authorBreedCache/authorCategoryCache），同一作者只查一次

**调用链路**：
```
PostController.getFeed(tab="recommend")
  → RecommendService.recommendPosts(userId, page, size)
    → computePostRecommendations(userId)
      → PostMapper.selectCandidatePosts(500)  // 候选集
      → PetMapper.selectByUserId(userId)       // 我的宠物
      → FollowMapper.selectFolloweeIds(userId) // 我的关注
      → PostLikeMapper.selectByUserId(userId)  // 我的点赞
      → 5维评分计算 → 排序 → 分页返回
```
